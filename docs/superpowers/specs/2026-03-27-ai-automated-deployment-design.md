# AI 自动化部署方案设计

## 背景

需要实现 AI 驱动的自动化部署能力，支持两种场景：
1. 个人项目：部署到云服务器（无跳板机）
2. 公司项目：部署到内网服务器（通过跳板机）

## 环境对比

| 维度 | 个人项目（验证环境） | 公司项目（目标环境） |
|------|---------------------|---------------------|
| 服务器类型 | 云服务器 | 内网服务器 |
| 网络访问 | 本地直接 SSH | 本地直接 SSH / CI/CD 需跳板机 |
| 跳板机 | 无 | 有 |
| 构建模式 | 本地构建 + 传输文件 | 本地构建 + 传输文件 |

## 方案一：CI/CD + SSH 部署

### 触发方式

支持多种触发方式：
- **标签触发**：`git tag v1.0.0 && git push origin v1.0.0` 自动部署
- **API 触发**：通过 GitHub API 或 CLI 触发，AI 可调用
- **手动触发**：GitHub Actions 页面点击运行

### 部署流程

```
触发 CI/CD → 拉取代码 → 构建后端(Maven) → 构建前端(Vite) → SSH 部署 → 健康检查
```

### 网络配置

**个人项目（无跳板机）：**
```
GitHub Actions ──SSH──> 云服务器
```

**公司项目（有跳板机）：**
```
GitHub Actions ──SSH──> 跳板机 ──SSH──> 内网服务器
```

配置方式：SSH ProxyJump
```bash
ssh -J user@jumpserver:port user@internal-server
```

### 关键配置文件

- `.github/workflows/deploy.yml` - CI/CD 流水线定义
- GitHub Secrets - 存储服务器凭证（加密）

## 方案三：AI 直接 SSH 执行

### 构建模式

采用**模式 B：本地构建 + 传输文件**

```
AI 在本地构建 ──SCP/RSYNC──> 上传到服务器 ──SSH 执行启动──> 运行
```

### 部署流程

```
本地构建后端 → 本地构建前端 → 打包构建产物 → SCP 上传 → SSH 执行部署脚本 → 返回结果
```

### 优势

- 不需要服务器安装 Maven/Node.js
- 不需要服务器访问 Git
- 本地构建速度快
- 完美适配内网环境

### 关键配置文件

- `deploy/config.local.yml` - 服务器连接配置（不提交 Git）
- `deploy/ai-deploy.sh` - AI 调用的部署脚本

## 验证计划

### 阶段一：验证方案三（AI 直接 SSH）

使用个人项目验证：
1. 配置云服务器 SSH 连接
2. 实现本地构建 + SCP 上传
3. 实现 SSH 执行部署命令
4. 测试完整部署流程

### 阶段二：验证方案一（CI/CD）

使用个人项目验证：
1. 配置 GitHub Actions
2. 测试标签触发部署
3. 测试 API 触发部署
4. 验证部署结果

### 阶段三：固化方案

1. 整理部署流程文档
2. 创建可复用的部署模板
3. 适配公司内网环境（跳板机配置）

---

## 安全考虑

1. **凭证管理**
   - SSH 私钥不提交到 Git
   - CI/CD 使用平台加密存储（GitHub Secrets）
   - 本地配置文件加入 .gitignore

2. **权限控制**
   - 使用专用部署账号，非 root
   - 限制部署账号的 sudo 权限

3. **审计日志**
   - CI/CD 自动记录每次部署
   - AI 部署可输出详细日志
